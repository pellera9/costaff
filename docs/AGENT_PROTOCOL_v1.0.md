# CoStaff Agent Protocol v1.0

> **Status**: Stable
> **Released**: 2026-05-04
> **Audience**: anyone writing an agent that plugs into a CoStaff
> deployment, either as a first-party plugin or as a third-party.

This document specifies how an external agent integrates with the
CoStaff manager agent. An agent that conforms to this protocol can be
registered with `costaff agent add`, will appear in the dashboard, will
be discoverable via the manager's tool list, and can use the four core
MCP tools to interact with users, tasks, and shared files.

The keywords **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, **MAY**
in this document are to be interpreted as in [RFC 2119].

[RFC 2119]: https://www.rfc-editor.org/rfc/rfc2119

---

## 1. Overview

A **CoStaff agent** is a containerised process that:

1. Exposes an [Agent-to-Agent (A2A)] HTTP endpoint so the manager can
   delegate work to it.
2. Ships a `costaff.agent.json` **manifest** declaring its name,
   version, and runtime requirements.
3. Optionally connects to the **manager core MCP** to send messages,
   record task history, and manage shared files via the four standard
   tools defined in §6.
4. Joins the shared `costaff_default` Docker network so other CoStaff
   components can reach it by container hostname.

The protocol is intentionally minimal — it specifies only what is
needed for plug-and-play interoperability. It does **not** dictate the
agent's internal architecture, language, framework, or model provider.

[Agent-to-Agent (A2A)]: https://github.com/google/A2A

---

## 2. Versioning

The protocol uses **semantic versioning** with two-component significance:

- A change to **MAJOR** indicates a breaking change. Agents declaring
  one MAJOR version MUST NOT be deployed against a manager supporting
  only a different MAJOR.
- A change to **MINOR** indicates additions that are backward-compatible
  with earlier minors of the same MAJOR. Agents declaring `1.0` MUST
  work against a manager supporting `1.0` or higher within MAJOR `1`.

`PATCH` is not used in protocol identifiers; the protocol is identified
by `MAJOR.MINOR` only (`"1.0"`, `"1.1"`, `"2.0"`, …).

Every agent manifest MUST declare:

```json
{ "protocol_version": "1.0" }
```

Manifests without `protocol_version` are treated as legacy and trigger
a warning at registration. Agents loaded with `costaff agent add
--strict` and missing `protocol_version` MUST be rejected.

The current manager supports **`"1.0"`** and is forward-compatible with
all `"1.x"` minor versions.

## 3. The manifest (`costaff.agent.json`)

Every agent project MUST contain a `costaff.agent.json` file at its
repo root. The manifest is declarative — it describes the agent, not
how to start it. Container start-up is governed by the agent's
`docker-compose.yaml` (or fragment generated by the CLI).

### 3.1 Required fields

| Field | Type | Description |
|---|---|---|
| `protocol_version` | `string` | Protocol version this agent targets. MUST be a `MAJOR.MINOR` string (e.g. `"1.0"`). |
| `name` | `string` | Lowercase kebab-case identifier. MUST start with `costaff-agent-`. Used as the agent's stable id everywhere (config, container hostname prefix, env var names). |
| `version` | `string` | The agent's own semver, independent of `protocol_version`. |
| `description` | `string` | One- or few-sentence summary. Surfaced in the dashboard and used by the manager when deciding whether to delegate. |
| `a2a_service` | `string` | Hostname suffix at which the A2A endpoint is reachable inside the Docker network. The manager dials `http://{a2a_service}:{port}`. SHOULD be `agent-<name-without-prefix>` (e.g. `agent-coding`). |
| `port` | `integer` | Port the A2A endpoint listens on inside the container. SHOULD be `8081` for first-party consistency. |
| `health_path` | `string` | Path on the A2A endpoint serving the agent card. MUST be `/.well-known/agent-card.json` per the A2A spec. |

### 3.2 Optional fields

| Field | Type | Default | Description |
|---|---|---|---|
| `env_required` | `string[]` | `[]` | Env vars that MUST be present at runtime. Registration via `costaff agent add` SHOULD prompt for missing ones. |
| `env_optional` | `string[]` | `[]` | Env vars that MAY be set; the agent SHOULD function without them. |
| `env_shared` | `string[]` | `[]` | Env vars the CLI may auto-fill from the user's main `.env` (e.g. `GOOGLE_API_KEY`, `LITELLM_*`). |
| `env_auto` | `string[]` | `[]` | Env vars the CLI populates automatically (e.g. `<NAME>_AGENT_MCP_URLS`). The agent reads them; nothing else writes them. |
| `model_env_var` | `string` | — | Name of the env var the agent reads to pick its model (e.g. `CODING_AGENT_MODEL`). The dashboard's per-agent model setting writes to this var. |
| `mcp_configurable` | `boolean` | `false` | If `true`, the dashboard exposes the per-agent MCP whitelist editor for this agent. |
| `mcp_env_var` | `string` | — | Name of the env var that receives the per-agent MCP URL bundle. Required when `mcp_configurable` is `true`. |
| `costaff_sdk_version` | `string` | — | A semver range describing the minimum CoStaff CLI / core version this agent needs (e.g. `">=0.2.0"`). The CLI MAY refuse to register an agent that requires a newer core. |

### 3.3 Reserved fields

Field names starting with `x_` are RESERVED for forward-compatibility
extensions. Implementations MUST preserve unknown `x_*` fields when
reading and rewriting the manifest. Agents MAY use `x_*` fields to
record their own metadata; the protocol assigns them no semantics.

### 3.4 Example

```json
{
  "protocol_version": "1.0",
  "name": "costaff-agent-coding",
  "version": "1.0.0",
  "description": "Sandboxed Python execution; returns code, files, and run results.",
  "a2a_service": "agent-coding",
  "port": 8081,
  "health_path": "/.well-known/agent-card.json",
  "env_required": ["GOOGLE_API_KEY"],
  "env_optional": ["GIT_TOKEN", "GIT_AUTHOR_NAME", "GIT_AUTHOR_EMAIL"],
  "env_shared": ["GOOGLE_API_KEY", "COSTAFF_AGENT_MODEL_PROVIDER", "LITELLM_MODEL_NAME", "LITELLM_API_BASE", "LITELLM_API_KEY"],
  "env_auto": ["CODING_AGENT_MCP_URLS"],
  "model_env_var": "CODING_AGENT_MODEL",
  "mcp_configurable": true,
  "mcp_env_var": "CODING_AGENT_MCP_URLS",
  "costaff_sdk_version": ">=0.2.0"
}
```

The machine-readable JSON Schema for the manifest is at
[`schemas/costaff.agent.json.schema.json`](./schemas/costaff.agent.json.schema.json).

## 4. The A2A endpoint

The agent MUST expose an HTTP server implementing the [Agent-to-Agent
protocol]. In Python with the Google ADK this means using
`adk_to_a2a_app()` (or the equivalent low-level wiring). In other
languages, follow the A2A spec.

[Agent-to-Agent protocol]: https://github.com/google/A2A

The endpoint:

- MUST listen on the port declared in `manifest.port`.
- MUST be reachable inside the `costaff_default` Docker network as
  `http://{a2a_service}:{port}`.
- MUST serve a JSON agent card at `health_path`. The agent card is
  consumed by the manager during discovery; its schema is the standard
  A2A agent card.
- MUST handle the standard A2A `POST /` request envelope and return
  the standard response envelope.
- SHOULD treat each incoming request as stateless from the agent's
  point of view; conversational state lives in the manager and is
  passed through, not in the agent.

If the agent has no sub-agents itself (the typical "leaf" case), it
MUST set `disallow_transfer_to_parent=True` and
`disallow_transfer_to_peers=True` on its root LLM agent. Without these
flags, certain models (notably Gemini) will hallucinate
`transfer_to_agent` calls and fail at runtime. See
[`skill/costaff-agent/A2A_SERVER_SKILL.md`] for the rationale.

[`skill/costaff-agent/A2A_SERVER_SKILL.md`]: ../../skill/costaff-agent/A2A_SERVER_SKILL.md

## 5. The shared workspace

CoStaff bind-mounts a single directory into every container at
`/app/data`. Inside this directory:

| Path | Owner | Purpose |
|---|---|---|
| `/app/data/<agent-name>/` | the agent | Private scratch space. Files go here first. |
| `/app/data/shared/` | manager | Read-write to all agents. Files placed here are visible to other agents and to channels for delivery. |
| `/app/data/uploads/` | channels | User-uploaded attachments are written here by channel processes. Agents MAY read. |

Agents MUST NOT write outside `/app/data/<agent-name>/` and
`/app/data/shared/`. They MUST use the `move_to_shared` tool (§6.3) to
publish a file from their private space to `shared/`, rather than
writing directly into `shared/`.

Hosts MAY map `/app/data` to any path — agents must not assume
anything about the path outside the container.

## 6. Core MCP tools

The CoStaff manager exposes a Core MCP server at
`http://costaff-mcp-costaff:8081/mcp` inside the Docker network.
Agents that opt in receive this URL through the env var named in their
manifest's `mcp_env_var`. The exposed tool surface for plugin agents is
restricted by the manager's `agent_mcp_filters` config (§7) to the
four tools below.

All four tools MUST be implemented by the manager; agents consume them.
This section is the agent-facing contract.

Detailed JSON Schemas are in [`schemas/tools/`](./schemas/tools/).

### 6.1 `send_message_now`

Send a message back to the user — text, optionally including absolute
paths to files in `/app/data/...` that should be delivered as
attachments.

| Parameter | Type | Required | Description |
|---|---|---|---|
| `channel` | `string` | yes | One of `telegram` / `discord` / `line` / `webchat`, or `default` to infer from `session_id` prefix (`tg_*`/`dc_*`/`line_*`/`web_*`). Substrings (`tg`, `dc`, etc.) also accepted. |
| `recipient` | `string` | yes | Either the user's hashed identity id or the platform-side raw id. The manager resolves the mapping. |
| `body` | `string` | yes | Message text. MAY include absolute file paths under `/app/data/`; the channel SHALL deliver each file as an attachment alongside the text. |
| `subject` | `string` | no | Reserved for channels that support a subject (e.g. email). Chat channels ignore. |
| `app_name` | `string` | no | ADK app name. Default `"costaff_agent"`. Agents normally omit this. |
| `user_id` | `string` | no | ADK user id. Normally provided by the manager when calling. |
| `session_id` | `string` | no | ADK session id. Used as fallback for `channel` inference and to scope the notification to a session. |

**Returns**: `string` — human-readable status. `"OK"` on success, an
`"Error: ..."` string otherwise.

**When to call**: when the agent has a result, partial progress, or
reminder it wants delivered to the user **outside** the normal A2A
return value. Most agents return their result as the A2A response and
do not need this tool.

### 6.2 `add_task_comment`

Append an immutable comment to a `ProjectTask`. Comments form the
permanent task history shown in the dashboard.

| Parameter | Type | Required | Description |
|---|---|---|---|
| `task_id` | `string` | yes | The task's id. |
| `user_id` | `string` | yes | The user the task belongs to. The manager rejects calls from unapproved users. |
| `author` | `string` | yes | Who wrote the comment — `"user"` or the agent's name (e.g. `"coding_agent"`). |
| `content` | `string` | yes | The comment body. MAY contain Markdown. |
| `comment_type` | `string` | no | One of `result` / `decision` / `issue` / `note`. Default `"note"`. |

**Returns**: `string` — `"Comment added to task <task_id>."` on success;
`"Task <task_id> not found."`, `"Error: <reason>"`, or the approval
denial string otherwise.

**When to call**: at the end of a delegated task to leave a permanent
audit trail; whenever the agent makes a non-trivial decision or hits an
issue worth recording.

### 6.3 `move_to_shared`

Copy a file or directory from the agent's private workspace to the
shared workspace, mirroring its relative path.

| Parameter | Type | Required | Description |
|---|---|---|---|
| `src_path` | `string` | yes | Absolute path under `/app/data/`. The manager rejects paths outside `/app/data/`. |
| `overwrite` | `boolean` | no | If `false` (default), refuse if destination exists. If `true`, replace. |

**Returns**: `string` — `"[OK] Copied <src> → <dst>"` on success;
`"[ERROR] ..."` or `"[WARN] ..."` otherwise.

**When to call**: after producing a deliverable, to make it visible to
other agents and to channels for attachment delivery. Do NOT write
directly into `/app/data/shared/`.

### 6.4 `list_data_files`

List files under a path inside `/app/data`, optionally filtered by a
glob pattern. Also accepts a single file path to test existence.

| Parameter | Type | Required | Description |
|---|---|---|---|
| `path` | `string` | yes | Absolute path under `/app/data/`. |
| `pattern` | `string` | no | Glob filter such as `"*.pdf"` or `"report.csv"`. |

**Returns**: `string` —
- For a directory: newline-separated absolute paths of matching files,
  or `"[EMPTY] ..."` if none.
- For an existing file: `"[EXISTS] <path>"`.
- For a missing file or path: `"[NOT FOUND] <path>"`.
- For a path outside `/app/data/`: `"[ERROR] ..."`.

**When to call**: to verify a file exists before reporting completion;
to enumerate produced artefacts before calling `move_to_shared`.

## 7. MCP tool whitelisting

The manager core MCP exposes ~40+ tools (`epics`, `stories`, `users`,
`reminders`, …). Plugin agents only need the four core tools in §6.
The manager applies a per-agent **whitelist** through the
`agent_mcp_filters` section of `config.json`:

```json
{
  "agent_mcp_filters": {
    "<agent_key>": {
      "costaff": [
        "send_message_now",
        "add_task_comment",
        "move_to_shared",
        "list_data_files"
      ]
    }
  }
}
```

Where `<agent_key>` is the agent's name with hyphens converted to
underscores (e.g. `business_analysis` for `costaff-agent-business-analysis`).
Agents with `mcp_configurable: true` SHOULD ship a default whitelist
covering only the four tools above. Operators can broaden the whitelist
through the dashboard if their agent genuinely needs more.

Without a whitelist the agent inherits the full Core MCP toolset.
This is **not** the recommended default — token bloat and tool
mis-selection both hurt accuracy.

## 8. Environment variables

The protocol reserves the following env-var **patterns** for use by the
CLI and the dashboard. Agent code SHOULD only read them; the CLI writes
them.

| Pattern | Owner | Meaning |
|---|---|---|
| `<NAME>_AGENT_MCP_URLS` | CLI writes | JSON map of MCP URL bundles the agent should connect to. The agent's MCP toolset loader reads this. |
| `<NAME>_AGENT_MODEL` | dashboard writes | The model id string the agent should use (e.g. `"gemini-2.0-flash-exp"`). |
| `COSTAFF_AGENT_MODEL_PROVIDER` | user-set | `gemini` or `litellm`. Switches which model implementation the agent loads. |
| `LITELLM_MODEL_NAME` / `LITELLM_API_BASE` / `LITELLM_API_KEY` | user-set | LiteLLM connection details. Agents using LiteLLM read these. |
| `COSTAFF_PREFERRED_LANGUAGE` | user-set | Default human language for responses (e.g. `"Traditional Chinese (繁體中文)"`). |
| `WORKSPACE_DIR` | container init | Resolves to `/app/data` inside containers; rarely needs to be overridden. |

`<NAME>` is the agent's name without the `costaff-agent-` prefix,
upper-cased and hyphens replaced with underscores
(e.g. `costaff-agent-business-analysis` → `BUSINESS_ANALYSIS`).

## 9. Lifecycle

The CoStaff CLI handles the agent lifecycle. Agents do not need to
implement any of these steps themselves; this section explains what the
CLI does and what the agent should expect.

| Stage | Triggered by | What happens |
|---|---|---|
| **Register** | `costaff agent add` | Manifest is read and validated. With `--strict`, all required fields and the JSON Schema are enforced. The agent is added to `external_agents` in `config.json`. |
| **Start** | `costaff start` (Tier 2) | Each enabled external agent's `compose-fragment.yaml` is brought up. The CLI passes through env vars from the user `.env`. |
| **Discovery** | manager startup | The manager reads `EXTERNAL_AGENTS_CONFIG` from its env and creates a `RemoteA2aAgent` wrapped in an `AgentTool` for each entry. |
| **Invocation** | manager LLM call | The manager LLM calls the agent's tool with a request string. The wrapper translates the call into an A2A `POST /` to `http://{a2a_service}:{port}`. |
| **Result** | A2A response | Whatever the agent returns becomes the tool result; the manager continues its turn. |
| **MCP whitelist update** | dashboard or `config.json` edit | The CLI regenerates `<NAME>_AGENT_MCP_URLS` and the agent must be **rebuilt** (not just restarted) to pick up the new env var. |
| **Remove** | `costaff agent remove <name>` | Containers stopped, `external_agents` entry deleted, source kept. |

## 10. Recommended folder layout (informative, not normative)

The protocol places **no requirement** on how agent source code is
organised inside its repo. An agent passing the §11 compliance levels
is conformant regardless of layout — `costaff agent add` does not look
at directory structure.

That said, for **Python agents using Google ADK** the CoStaff project
publishes a canonical layout at
[`costaff-agent-template`](https://github.com/costaff-ai/costaff-agent-template).
New Python agents SHOULD follow this layout unless they have a specific
reason not to. Following it gives:

- Consistent file paths across the ecosystem (faster code review,
  easier knowledge transfer between agents).
- AI assistants writing new agents pick up the structure
  automatically — no need to re-explain conventions per repo.
- The CLI's `costaff agent add --github <url>` tooling assumes this
  layout when generating compose fragments and per-agent env files.

The template structure (top-level files plus the most relevant
subdirectories):

```
costaff-agent-<name>/
├── costaff.agent.json       # the manifest (§3)
├── docker-compose.yaml      # standalone deploy + bind mounts
├── agent/
│   ├── agent.py             # LlmAgent definition
│   ├── agent_a2a.py         # to_a2a() entry point
│   ├── Dockerfile
│   ├── requirements.txt
│   ├── instruction/         # system prompts
│   ├── models/              # gemini / litellm provider switch
│   ├── mcp_toolsets/        # connection to manager core MCP
│   ├── skills/              # agent-specific prompt templates
│   └── sub_agents/          # nested A2A delegation (optional)
└── mcp/                     # OPTIONAL: agent's own MCP server
    ├── server.py
    ├── Dockerfile
    └── tools/
```

Non-Python agents (Go, Node, Rust, …) and intentionally minimal Python
agents MAY use any layout that meets §3 / §4 / §6.

## 11. Non-goals

This protocol explicitly **does not** specify:

- **Agent internals.** Language, framework, model provider, prompt
  layout, sub-agent topology, retrieval strategy — all out of scope.
  See §10 for the recommended Python layout, but it is purely advisory.
- **The manager's prompting or planning.** How the manager decides
  which agent to delegate to is implementation-specific to the manager.
- **Streaming protocols.** The current manager treats A2A calls as
  request/response. Streaming is a future protocol extension.
- **Authentication between agent and manager.** Today both run on the
  same Docker network and trust each other implicitly. A future minor
  may add bearer-token auth declared in the manifest.
- **Cost accounting.** Per-agent model usage tracking is a manager
  feature, not part of the protocol.

## 12. Compliance levels

| Level | What it means |
|---|---|
| **Minimal** | Manifest has the required fields in §3.1; A2A endpoint serves the agent card; agent works. |
| **Schema-valid** | Manifest passes the JSON Schema in `schemas/costaff.agent.json.schema.json` — every field, including optionals, has the right type. `costaff agent add --strict` enforces this. |
| **Tool-conformant** | Agent uses the four core tools in §6 only via the manager core MCP, not by re-implementing them or calling them directly. Whitelist in §7 applied. |
| **Future-proof** | Honors `x_*` extension fields, declares `costaff_sdk_version`, follows the env-var conventions in §8 verbatim. |

New Python agents SHOULD start by forking
[`costaff-agent-template`](https://github.com/costaff-ai/costaff-agent-template):
it ships a working agent at the **Schema-valid + Tool-conformant** level
and uses the recommended layout from §10. Layout aside, **layout is not
part of compliance** — an agent in any language with any directory
structure that meets §3 / §4 / §6 is conformant.
