② 關鍵原則

步驟規則
分類一輪訊息 = 一個動作。iteration / 修改也視為新 task，不可口頭承諾。
派工create_project_task 必須接 update_task_queue（同一 turn）— Principle 0
平行禁止 絕不平行 invoke 多個 sub-agent — Principle 3
鏈式Coding 完 → callback → Manager 決定要不要接 BA → 再 create+queue 一次。
回報Sub-agent 收到 [PROGRESS_CONTEXT] 才 call send_message_now。Prefix [Agent名]、無 emoji、講重點。
結果Sub-agent 一律以 [RESULT_START]...[RESULT_END] 包裹最終回覆。
附件dispatch_notification 從結果文字 grep file path → 逐檔 sendDocument。
路徑Coding 寫到 caller 指定路徑逐字照寫，禁止偷塞 outputs/ 子層。

③ 三條典型路徑

A. 純資料查詢

例：「請查最近一週台北 PM2.5」

User → Manager
     → create+queue(Twinkle Hub)
     → 執行
     → callback
     → Manager 回覆 ＋附件

B. Code → Report 鏈

例：「分析 wine dataset 並出 PDF」

User → Manager
     → create+queue(Coding)
     → 執行 → callback
     → Manager
     → create+queue(BA，帶上一階段路徑)
     → 執行 → callback
     → Manager 回覆 ＋ PDF

C. Iteration / 修改

例：「把剛剛報告的圖改成長條圖」

User → Manager
     → 判定 iteration
     → 仍然 create+queue 真實 task
       （禁止「我會去改」幻覺）
     → 執行 → callback
     → Manager 回覆 ＋新檔案

⑤ 指派任務（Task Dispatch）詳解

5.1 Task 狀態機

stateDiagram-v2
    [*] --> backlog: create_project_task
    backlog --> queued: update_task_queue
（同 turn 必接）
    backlog --> cancelled: 用戶取消 / Manager 改主意
    queued --> running: Executor 抓走
    running --> done: RESULT_START...END 收到
    running --> failed: A2A 異常 / timeout / 3 次重試失敗
    failed --> queued: Manager 決定 retry
    done --> [*]: SYSTEM_CALLBACK 回 Manager
    failed --> [*]: SYSTEM_CALLBACK（帶 error）回 Manager

5.3 Spec 該帶什麼（Manager 寫給 sub-agent 的 payload）

欄位必要說明
task_description必要一句話講做什麼（user-readable）
inputs必要上游檔案路徑 / 參數 / 用戶補充
expected_outputs必要預期產出路徑（Coding 必須逐字照寫）
success_criteria建議成功判定（測試通過 / 檔案存在 / row 數正確）
progress_context自動{user_id, channel, session_id} — executor 注入，sub-agent 用來 callback 進度
prior_resultschain 才有上一階段的 RESULT 摘要（路徑、key findings）

5.4 三條典型派工模式

模式Manager 動作callback 後動作
單 agent（PM2.5） create+queue(twinkle) dispatch_notification → user
雙 agent chain（wine→PDF） create+queue(coding) callback → create+queue(ba, 帶 coding 路徑) → callback → notify
Iteration（改昨天的圖） 仍然 create+queue（不可口頭承諾） 同單 agent 流程

5.5 三個最容易出錯的點

錯誤後果
只 create 不 queue task 卡在 backlog，executor 永遠不抓 → user 看不到任何進度 → 像 Manager 失憶
平行 create 兩個 agent callback 順序不可控，spec 路徑漂移
Spec 路徑不一致 Coding 寫到 A 路徑、BA 去 B 路徑找 → BA 報 file not found（這就是為什麼 Coding 要逐字照 spec）
