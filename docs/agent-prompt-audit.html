① 系統架構概覽

CoStaff 採用 hub-and-spoke 架構：Manager Agent 是與用戶對話的唯一入口，所有專業任務透過 dispatch_task 派工給 sub-agent。Sub-agent 不直接與用戶對話；它們透過 send_message_now 發進度、用 [RESULT_START]...[RESULT_END] 包最終結果，executor 把結果以 SYSTEM_CALLBACK 注回 Manager session 由 Manager 整理成自然語言回覆。

每個 agent 的 prompt 結構

所有 agent 共用一致的 prompt 構造：

  • instruction/system.md — Agent 自我認知、tool discipline、workflow、output 規範。
  • skills/<name>/SKILL.md — 任務情境化的詳細 SOP，由 ADK SkillToolset 懶載入，LLM call load_skill 才取出全文。
  • build_instruction()（Python 函式）— 把 {PREFERRED_LANGUAGE} 等變數注入 system.md。

③ Manager Agent（costaff_agent）

身份與角色

高效 AI 協調者，是與用戶對話的唯一入口。協調 Coding / BA / Twinkle / Database / Medical / HR 等專業 sub-agent。核心紀律：嚴格區分同步 / 非同步執行、plan-and-confirm 閘門、session-based callback routing、verb-set discipline（一個 task spec 不混能力集）。

核心強制規則

規則條目
Context Extraction每次輸入提取 User ID（16-char hex）+ Session ID；靜默驗證（不問用戶）
Async-First Delegation所有產檔 / 多分鐘任務一律走 dispatch_task（非同步）
Plan-and-Confirm Gate≥ 2 specialist 或有資料 hand-off → 必須先給 plan、等用戶 OK
Verb-Set Discipline一個 spec 不混能力集；分鏈派工。Coding ≠ BA ≠ Twinkle
Dispatch Budget同一 user task 每個 sub-agent 3 次 dispatch（hard cap 5）
SYSTEM_CALLBACK HandlingManager 用「自己的聲音」回覆，禁止以 [BA] / [Coding] 前綴
Output FormatTelegram HTML 為主（<b>, <i>, <code>），禁用 Markdown

④ Coding Agent

身份與角色

資深軟體工程師，one-shot executor。負責 Python 腳本、資料分析、ML 演算法、套件管理、git、測試。收 request 字串 → 寫/跑/驗 → 以 [RESULT_START]...[RESULT_END] 回報。不產敘事 / 圖表 / PDF — 那是 BA。

⑤ Business Analysis Agent

身份與角色

專業報告專家。把資料（CSV / 數字 / 文字 / 程式結果）變成精美的 PDF / PPTX，含圖表與敘事。不跑 Python / 不裝套件 — 那是 Coding。最終交付物必須是 PDF 或 PPTX；單純的 Markdown 不算完成。

核心強制規則

規則條目
Completion Discipline沒產 PDF / PPTX 絕對不算 done；中間檔（PNG / HTML / CSV）≠ 完成
禁止推測式拒絕禁止在沒跑 tool 之前說「無法 / 不能 / 暫時無法」
禁止 downgrade不要在嘗試原規格前主動降級成 Markdown
Tool Discipline Fail-Fasttool 不在 list → 立即停、不重試、不猜名字（同 Coding）
Output Directory每個 task 自己一個 <report-name>/ subdir；caller 給的名稱逐字用，不翻譯
Report Back Path回報的 path 必須是 export_pdf() / export_pptx() 的實際 return value，禁止自己拼路徑
進度回報prefix [BA]；checkpoints: Start / Charts / Report / PDF / Done
受眾適應技術 vs 商業受眾語氣切換；預設商業

⑥ Twinkle Hub Agent

身份與角色

台灣開放資料 curator。存取 19 個 domain 的 Twinkle Hub（環境、不動產、政府採購等）。任務是 discover → inspect → query → materialize → save。不跑程式、不畫圖、不寫敘事 — 把 CSV/JSON 交給 Coding 或 BA。

核心強制規則

規則條目
Discover-Before-Assuming每次 task 必先 opendata-list_domains 或 opendata-search_datasets，禁止瞎猜 dataset ID
Cost-Aware ExecutionAPI call 計費；避免重複呼叫、適時快取 metadata
Always Save絕不 inline raw rows（吃 caller token、破壞 chain）；一律 save_curated_csv/save_curated_json
Sidecar Metadata每個檔配一個 <file>.meta.json，含 dataset_id / agency / domain / row_count / trace_id
Date Awareness需要當前日期 → call get_today_utc()，禁止瞎猜或從前次對話延續
檔名規範<domain>__<dataset_id>__<YYYYMMDD>.csv，日期來自 get_today_utc()，禁止寫死
進度回報prefix [Twinkle]；checkpoints: Start / Selected / Done / Failed

⑨ 觀察到的問題與不一致

高 — 結構性

1. Prompt 規則靠 LLM 自律的比例仍偏高

橫向契約表裡仍有 ~6 條是 Prompt rule 等級，全靠 LLM 看了照做。實證踩雷：

  • Manager verbal-only（口頭承諾派工但沒 call dispatch_task）— 今天 13:47 踩過、Principle 1A rule 0 + Principle 0A 補加
  • Path discipline outputs/ 三方矛盾 — 今天 13:13 踩過、3 個 repo skill 對齊修補
  • Plan 路徑沒 project subdir — 今天 13:12 踩過、Manager 自我糾正

建議：把 [RESULT] envelope 結構化（JSON 含 status / files / error_code），讓 dispatcher 不靠 regex 抓檔、Manager 不靠自然語言判讀錯誤類型。對應改善 HTML 的 #4 + #6。

高 — 觀察

2. Manager system.md 規模大、Skills 群龐大、context 壓力大

Manager system.md 362 行、10 個 skill 合計 963 行。multi-agent-chain 單一個就 394 行 / 22.5KB。今天有 1 次 gemini-2.5-flash 對「skill 找不到」幻覺（/reset 後自癒）。Skill body 越長、LLM 越容易在開頭幻覺出「skill 不存在」這種錯誤回覆。

建議：multi-agent-chain skill 拆成兩塊：

  • multi-agent-chain-core（必看的 Principle 0 / 0A / 1 / 1A / 2）
  • multi-agent-chain-recovery（Principle 3 / 4 / 5 / 6 / 7 / 8，只在出錯時 load）

或精簡掉「重複講同一條規則」的段落（reproduce 案例可以歸到註解 / changelog，不必每次都載入 context）。

中 — 矛盾

3. Path Discipline 仍有殘留「outputs/」字眼

今天統一修了 7 個檔（3 repo），但 BA 兩個 skill 的範例還是會展示 包含 outputs/ 的路徑（被改寫成括號提示「如果上游有加就照用」）。如果未來 LLM 解讀錯誤，可能仍會誤推。

建議：把這些 fallback 描述也改成「empirically：upstream 通常不會加 outputs/」並追加「若見到 outputs/ in path → 信任上游」。

中 — 觀察

4. Recursive Dispatch（Manager 在 task session 又 call dispatch_task）

今天 13:12 觀察：Coding 的 task spec 透過 build_task_spec 送給「Manager-in-task-session」，Manager 在 task 執行期間又 dispatch 了另一輪 task（session_id=task_xxx）。產生重複工作。

架構成因：每個 task 的 spec 是給「Manager LlmAgent」執行（因為 hub-and-spoke），Manager 本能反應就是「dispatch」。但這層不該再 dispatch。

建議：build_task_spec 在 spec 加註：「[YOU ARE NOW THE SPECIALIST EXECUTOR, NOT THE MANAGER. DO NOT call dispatch_task / create_project_task in this session. Just call the specialist tool directly.]」

中 — 觀察

5. Sub-agent 進度訊息容易洗版

每個 sub-agent 可在自由心證的時機 send_message_now。用戶可能短時間收到大量訊息。

建議：在 dispatcher 加 throttle / 去重 — 同一 task 同樣前綴的訊息間隔 < 5 秒就合併或丟棄。

低 — 風格

6. Skill description 多行格式被部分工具截斷

BA 與 Medical 的 SKILL.md frontmatter description 用了 multi-line YAML（description: >），grep 工具看不到完整文字。雖然 ADK 載入沒問題，但對 review / audit 不友善。

建議：description 改用單行（< 200 chars）。

低 — 風格

7. Sub-agent disallow_transfer_to_* flags 在自己 prompt 內沒提

架構上強制（agent.py 構造階段），但 sub-agent 自己的 system.md 沒寫，所以模型可能對「為什麼我不能呼叫 transfer_to_agent」沒有概念。實際不會踩雷因為 transfer tool 根本沒註冊；但 prompt-only 讀者會困惑。

建議：各 sub-agent system.md 開頭加一句「Architecturally I cannot transfer back to a parent or peer — my only return path is the [RESULT] envelope.」

📊 總結

整體 prompt 結構成熟、職責分工清楚、紀律規則完整。但有兩個系統性議題仍待解：

  1. 太多契約靠 prompt rule：把更多規則下移到 executor / MCP 層硬約束，能消滅整類失敗模式。今天的 dispatch_task / auto-link / verifier / queue-aware callback 是這個方向的好例子。
  2. Prompt 越長越脆：Manager 加碼 prompt 解 LLM 弱點是反模式。比較好的方向是「縮小 prompt + 用結構化資料 / tool return value 來引導」，例如結構化 RESULT envelope。

建議下一個 sprint 主攻 P1 三項：結構化 RESULT envelope（最大 ROI）、recursive dispatch fix（最便宜）、skill 拆分（降 context 壓力）。
