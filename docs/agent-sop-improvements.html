CoStaff Agent 流程改善對比

現況 vs 建議改法 — 把靠 LLM 自律的「軟約定」換成 executor / MCP 層的「硬約束」

① 核心觀察

目前流程的所有「契約」都靠 LLM 自律（Principle 0、平行禁令、Spec 路徑、結果格式…）。一旦換 model、context 變長、cache 過期、被壓縮——任一條規則都可能失守。今天 PM2.5 task 卡在 backlog 就是 Principle 0 失守的實例。

改善目標：把契約從 prompt rule 下移到 executor / MCP rule，幻覺空間 = 0。

② 6 大問題總覽

❌ 現況

Coding: "[RESULT_END] done, file at X"
Executor: state=done, callback Manager
Manager: dispatch(ba, input=X)
BA: read_file(X) → FileNotFoundError
BA: 我做不到

問題：
  - 錯誤被延後到 BA 才暴露
  - 用戶等了 30+ 秒才看到失敗

✅ 建議

Coding: result.files = [X]
Executor:
  for f in expected_outputs:
    if not os.path.exists(f):
      state = failed
      error_code = OUTPUT_MISSING
      reason = f"declared {f} but not on disk"
  else:
    state = done

callback 帶錯誤回 Manager
  → Manager 立刻 retry / 換 agent / 告訴 user
6 Error Code Enum（不是自然語言）
嚴重度 中 · 工作量 ~1d · Manager retry 決策可機械化

❌ 現況

Sub-agent 失敗回：
"I cannot complete this task. The spec
 asks for X which requires Y. That is
 the responsibility of Z, not mine."

Manager LLM：
  讀自然語言判斷
  → retry? 換 agent? 告訴 user?
  → 判錯率 > 0

✅ 建議

error_code enum:
  TOOL_NOT_AVAILABLE  → Manager 換 agent
  WRONG_AGENT         → Manager 換 agent
  INPUT_MISSING       → 回問 user
  TIMEOUT             → Manager retry (1 次)
  PERMISSION_DENIED   → 告訴 user
  UNKNOWN             → 告訴 user + 附 raw

Manager：用 dict lookup 決策
  → 零判錯

③ 建議落地順序
